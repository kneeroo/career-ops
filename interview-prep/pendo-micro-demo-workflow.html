Before vs after — what Pendo gives you vs what we shipped

Pendo's Settings → Subscription Settings hands you a single snippet that combines two things: the agent loader (IIFE that injects pendo.js from their CDN) and a template pendo.initialize() call with placeholder visitor / account fields. You're expected to replace the placeholders with real values from your auth context.

Because Lumen is a React/Next.js App Router app, we split the snippet across two locations for proper lifecycle handling, then enriched the initialize call with realistic demo-patient metadata.

What got implemented

1. Structured visitor & account identity

Replaced the anonymous random ID with a real-looking demo visitor (id: demo-patient-001, email, full name, role, state/country, is_demo_user: true) so segments like "Time Since First Visit < 30 days" actually work. The localStorage random ID was preserved as anonymous_session_id metadata so multiple browser sessions still register as distinct for Paths.

Account expanded with plan_level, region: ANZ, industry: Healthcare — useful for segment rules and visible in Pendo Settings on camera.

2. App Router page-load tracker (the critical fix)

Created pendo-page-tracker.tsx, a client component that listens to usePathname() and calls pendo.pageLoad() on every route change.

This is non-obvious and breaks most naïve Pendo installs on Next.js App Router. The App Router does client-side navigation without firing the URL events Pendo's auto-tracker depends on, so without this hook, Paths and Funnel reports only see the first page per session. Fixing this matters because the whole demo is built around a Paths report.

3. Resilient data-pendo-id selectors

Added data-pendo-id attributes to the key tap targets — every sidebar nav button (nav-home, nav-visits, nav-actions, etc.), the "Ask a question" button on the patient portal, and the existing View Summary button (open-patient-portal).

Feature tagging now uses stable selectors that survive classname or button-text changes — the pattern Pendo's own docs recommend for production installs. This is the same gotcha I deliver on camera in Segment A.

4. Env-var gating preserved

The snippet stays gated on NEXT_PUBLIC_PENDO_API_KEY. Without the key, the component returns null so local dev keeps working. The key (the App ID from Pendo's Subscription Settings) sits in Vercel env vars only — never committed.

Important: tagging happens BEFORE recording

DOM fumbling on camera looks unprofessional. Tag everything once, then walk through the pre-tagged list on camera.

Pre-recording: Tag Pages (Home + My Visits as the funnel source)

  1. Product → Pages → Tag Pages (top-right button)
  2. Tag Home → URL rule /home → Save
  3. Tag My Visits → URL rule /visits → Save

Why these two? The story is the funnel: Home → My Visits → View Summary. Tagging the source pages lets the Paths report show where patients drop off, not just whether they hit the destination.

Pre-recording: Tag Feature (the View Summary button)

  1. Product → Features → Tag Features (top-right)
  2. Enter URL of the Page hosting the View Summary button (/visits)
  3. Choose Tag side-by-side
  4. Click the View Summary button in the live preview
  5. Use up/down arrows top-right of Studio to walk DOM tree if needed
  6. Under Feature Element Matching, accept Suggested Match
  7. Name: "View Summary" → Save

On-camera (the 15 seconds the viewer actually sees)

  1. Click Product → Pages in the left nav; pre-tagged Home + My Visits visible in the list
  2. Click Product → Features; pre-tagged View Summary visible
  3. Narrate over the navigation (see "What you say" above)
  4. Move on (don't dwell)
What viewer sees

Pendo Pages list with Home Page and My Visits visible, then Features list with View Summary visible. Clean and pre-built.

Honest gotcha to mention on-camera (optional bonus)

"CSS-class selectors break when devs change classnames. In a real onboarding I'd walk the dev team through Pendo's data-pendo attribute pattern. The selector I ended up with for this Feature is a[href^="/patient-portal?visit="], which happens to be a stable href-prefix match — it survives any visit ID and any button-text change. I've named the Feature 'View Summary' to match the button text for the micro demo, but in production I'd push for intent-based naming like 'Open Patient Portal' so the Feature still makes sense if marketing renames the button next quarter."

  6. Date range: Last 30 days → Run → Save the report so you can reopen instantly on camera

What this Guide does: puts a tooltip on the View Summary button on the My Visits page, nudging patients within their first 30 days on the platform. This is the discoverability fix for new patients who haven't discovered the feature yet.

Important: you're NOT creating the tooltip live on camera

8 build steps in 15 seconds is impossible. You build the Guide as a Draft once, before recording. On camera you just open the pre-built Guide Designer view and walk through what's already configured.

Pre-recording: build the Guide as a Draft (do this the day before, takes ~2 minutes)

  1. Guides → + Create Guide (top-right)
  2. Choose Layout → Tooltip from gallery
  3. Step 1 panel: Page Location = My Visits (so the tooltip fires when patients land on the visits list, before they need to find the button themselves)
  4. Anchor element: click the View Summary button in the live preview; the tooltip pins to it
  5. Edit copy: "New: read your visit summary in plain language."
  6. Activation card: Automatic on page load, or Automatic with Badge
  7. Segment: select "Time Since First Visit: < 30 days" (Pendo built-in time-based segment using visit metadata; targets new patients who may not have discovered the View Summary feature yet)
  8. Schedule: Start date = today → Save Draft (do NOT Publish)

On-camera (the 15 seconds the viewer actually sees)

  1. Click Guides in the left nav; View Visit Summary Tooltip is visible in the Draft list
  2. Open View Visit Summary Tooltip (one click)
  3. Narrate while the Designer loads (see "What you say" above, peppered with the "for the purpose of this demo" framing)
  4. Briefly hover-highlight the Segment card on the right panel so the viewer sees the targeting rule
  5. Move on (don't dwell)
