A
Tag Feature + Pages
Script timing: 0:22 – 0:34 · 12 seconds

Tag Pages

  1. Product → Pages → Tag Pages (top-right button)
  2. Enter URL → name Page (e.g., "Patient Portal")
  3. Visual Design Studio loads → confirm URL rule → Save

Tag Feature

  1. Product → Features → Tag Features (top-right)
  2. Enter URL of the Page hosting the patient-portal button
  3. Choose Tag side-by-side (better for demo)
  4. Click the patient-portal button in the live preview
  5. Use up/down arrows top-right of Studio to walk DOM tree if needed
  6. Under Feature Element Matching, accept Suggested Match
  7. Name: "Open Patient Portal" → Save
What viewer sees

Pendo Visual Design Studio with the Lumen Health app rendered in main panel, right-side panel showing CSS selector rule, highlight box around tagged button.

Honest gotcha to mention on-camera

"CSS-class selectors break when devs change classnames — in a real onboarding I'd walk the dev team through Pendo's data-pendo attribute pattern. Notice the visible button text is 'View Summary' but I'm naming the Feature 'Open Patient Portal' — that's intentional. Feature names should describe user intent, not button copy. If marketing renames the button next quarter, my Path report doesn't break."

B
Paths report
Script timing: 0:34 – 0:56 · 22 seconds
  1. Analytics → Reports → Create report → Path
  2. Activity source: Select event → "Open Patient Portal" Feature click
  3. Anchor direction: dropdown → switch from "Starting from" to "Leading to"
  4. Segment: Everyone (or "Active in last 30 days")
  5. Date range: Last 30 days → Run
What viewer sees

Sankey-style flow diagram with Page-to-Page transitions, percentages on each branch.

Honest call-out — DO NOT fabricate stats

"10-second bounce" is NOT a native Paths metric. Read real numbers off the chart. If you need time-to-bounce, pair Paths + Funnel.

Recommended framing: "Paths shows me where they fall off — looks like [READ ACTUAL %] never reach the portal. I'd pair this with a Funnel for time-to-convert if I wanted the bounce-time number."

Tooltip Guide

  4. Anchor element: click the patient-portal button — tooltip pins to it

NPS survey
Script timing: 1:14 – 1:22 · 8 seconds
  1. NPS → + Create NPS Survey → opens 4-step wizard
  2. Step 1 Content: customize 0–10 question + follow-up text
  3. Step 2 Segment: "Create new segment" → rule: Feature "Open Patient Portal" used ≥ 1 time in last 14 days
  4. Step 3 Schedule: Start now, recurring every 90 days
  5. Step 4 Review → Save Draft
Strongest single line in the demo

"Pendo's own guidance is to target NPS at visitors with first visit >30 days ago. I'm loosening that here for early-signal feedback right after a feature launch — that's the trade-off I'd discuss with the customer's CS lead."

Reading and intentionally diverging from a Pendo-published best practice = the most senior thing you can do in 90 seconds.

90-Second Script

Total runtime target: 92 seconds. If over, cut "Different fix, visible in the data" line in Paths.

Open 12 seconds

Hey Pendo team, I'm Niroo, applying for the Senior Customer Engineer role in ANZ. The thing I love about Pendo is that it's the flight recorder for product decisions. Most product teams are flying blind on adoption. Pendo gives them the black box. My job as a CE is to make sure customers know how to read it.

Customer Setup 10 seconds
[Switch to Pendo workspace, "Lumen Health" workspace name visible]

Quick scenario: Lumen Health is a post-visit patient companion — patients log in to read the plain-language summary of their consultation. 8,000 MAU. They bought Pendo because adoption of the summary view is flat, and the VP of Product needs a number by end of month.

Tag 12 seconds
[Show Product → Features view with "Open Patient Portal" Feature visible]

Day one, I tag the summary button as a Feature — I'll call it "Open Patient Portal" in Pendo regardless of the visible button text — plus the My Visits and Patient Portal pages. Two minutes of setup, no engineering ticket. From here every interaction is on the record.

Paths 22 seconds
[Open Paths report, "Leading to Open Patient Portal", Last 30 days]

By day seven, I run a Path report leading to the portal. [READ ACTUAL %] of users never reach the feature. The ones who do, drop off here [point at node]. So the VP doesn't have an adoption problem — they have a discoverability problem. Different fix, visible in the data.

Guide 18 seconds
[Open Guide Designer with tooltip on My Visits]

So we ship a Guide — a one-line tooltip on the My Visits page that nudges users toward the summary on their next session. Targeted at users who haven't opened a summary in seven days. No code, no deploy, live in an hour. Two weeks later we re-run the Path report. The lift becomes the number the VP takes to their board.

NPS 10 seconds
[Show NPS wizard, Step 2 Segment with feature-usage rule]

Then we trigger NPS to the users who actually used it. Pendo's guidance says target visitors with 30+ days tenure — I'm loosening that here for early-signal feedback. That's the trade-off I'd discuss with the customer's CS lead. The loop closes.

Close 8 seconds

Picture one of your existing ANZ customers — the one whose CSM is asking what to show in the next QBR. This is the motion I'd run for them on Monday. Looking forward to talking.
