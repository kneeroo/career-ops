How Pendo is configured on Lumen Health

The business-speak version. Use this when an interviewer asks "what did you actually do?" Walks through six setup decisions and what each one means for the business.

The headline

"Lumen Health is fully instrumented with Pendo from day one. Every patient session is identified, every page view is captured, and every key button is tagged, so the product team can answer 'are patients actually using this?' without writing a line of SQL."

What's been set up

1

Pendo is live on every page

The Pendo agent loads automatically across the whole app, switched on by a single environment variable. If we turn the key off, the app keeps working; analytics just go dark. Safe by default.

2

Every visitor has a rich identity

Instead of anonymous "user 1234s," each patient comes with structured attributes: name, email, age, state, postcode, role, and how long they've been on the platform. That means in Pendo we can answer questions like "are patients over 50 in regional states using the medications page?" on day one.

3

The demo is populated with realistic people

I built an eight-person fictional patient roster: mixed ages, every Australian state, sign-up dates staggered across two years. Every new browser session is assigned a different patient, so when a hiring manager opens the demo, Pendo's dashboards look alive (multiple users, segments, paths) instead of one lonely visitor clicking around.

4

Page tracking actually works on a modern Next.js app Kill shot

This is the bit most teams get wrong. Next.js 16's App Router doesn't tell Pendo when the page changes, so without a fix you only ever see the landing page in your funnels. I wrote a tiny listener that tells Pendo every time the user navigates, so Paths, Funnels, and Retention reports are accurate end-to-end.

5

Key buttons are pre-tagged for the analyst

The sidebar nav links, the "Open your most recent visit" button, and the "Ask a question" CTA all carry stable Pendo identifiers. That means a PM or analyst can build a funnel in Pendo's UI in minutes, no engineering ticket required, and the tags don't break when designers change the styling.

6

Production hygiene is built in

Every demo patient and the demo account are flagged is_demo_user: true. When this goes to a real customer, filtering the synthetic traffic out is one checkbox, no data clean-up project required.

What this means for the business

CapabilityWhy it matters
Patient segmentation on day oneProduct decisions based on who is using what, not gut feel
Working funnels in the App RouterReliable conversion data, no "we'll fix tracking next sprint"
Self-service tagging for analystsFaster insight cycles, less engineering bottleneck
Realistic demo dataStakeholders see what good looks like instead of empty dashboards
Demo-flagged usersClean handover from prototype to production analytics

The one-liner if asked in an interview

"I treated Pendo as a first-class part of the product, not a tag I sprinkled on at the end. The visitor schema is designed for segmentation, the App Router quirk is solved, and the demo is seeded with realistic users so the analytics actually tell a story."

Delivery note: three clauses, easy to remember under pressure. Works as a wrap-up after walking through the six setup items, or as a standalone answer if the interviewer doesn't want the full tour.

T3. "When would you use the Track API instead of the snippet?"
Server-side events (signed Track API with HMAC), backend-fired events that don't have a UI surface, batch backfills. Snippet is for everything UI-driven.
T4. "How do Pendo segments evaluate? Why doesn't a published guide always fire immediately?"

T5. "Walk me through the difference between a Page and a Feature in Pendo"
Page = URL / route rule. Feature = element click matched via selector. Pages tag where users go; Features tag what they do.
T6. "How would you debug 'no events firing'?"
