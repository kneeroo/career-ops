Walk me through how you installed Pendo on a real app.

"I built a Next.js healthcare app, Lumen Health, and installed Pendo on it end to end. There are really four pieces. First, get the agent on the page, the Pendo snippet, a piece of JavaScript that loads Pendo's tracking script; I put it in a component that loads across the whole app and gated it behind an environment variable, so with no API key Pendo just quietly doesn't load. Second, tell Pendo who the user is, it needs a visitor ID to attribute anything. Third, fix page tracking, my app is a single-page app so navigating swaps content without a full reload, and Pendo's auto-tracker listens for page loads, so I added a piece that watches the URL and calls pendo.pageLoad() on every route change, which is what makes the multi-step funnel and path reports work. Fourth, tag the things that matter, I added stable data-pendo-id attributes onto the key buttons and nav links so feature tagging grabs onto something reliable instead of a CSS class that changes every build."

The architecture, three components wired app-wide via providers.tsx:

pendo-snippet.tsxLoads the agent + identifies the visitor/account.
pendo-page-tracker.tsxFixes SPA page tracking (the App Router gotcha).
data-pendo-id attributesStable selectors on key UI elements for click tracking.

1 · Agent install

  • Standard Pendo snippet injected via Next's <Script strategy='afterInteractive'>, pulling the agent from cdn.pendo.io/agent/static/<apiKey>/pendo.js.
  • API key from process.env.NEXT_PUBLIC_PENDO_API_KEY (the NEXT_PUBLIC_ prefix Next requires). If unset, the component renders null and Pendo never loads, gated behind config, safe to ship without a key.
  • A global Window.pendo type is declared so initialize/pageLoad/identify are typed.

2 · Visitor / account identification

  • In a real production install the visitor ID comes from your auth provider, e.g. Clerk's useUser() gives you the real user ID and email, and the org becomes the account.
  • For this demo I went the other way on purpose: an 8-person PATIENT_ROSTER of fictional AU/NZ patients, each with state, postcode, age and staggered created_at dates, so geo segments, time-on-platform segments, Paths and Funnels have real variety.
  • getSessionPatient() randomly assigns one per browser session, persisted in sessionStorage (reload keeps the visitor; a new tab gets a fresh one).
  • pendo.initialize() runs in a useEffect, passing that patient as visitor (id, email, name, role, geo, age, is_demo_user) plus a fixed account (lumen-health-demo, plan demo, region ANZ, industry Healthcare).

3 · SPA page tracking

  • Next's App Router does client-side navigation without full page loads, which Pendo's auto-tracker doesn't catch, so without this it only sees the entry page per session.
  • pendo-page-tracker.tsx watches usePathname() and calls window.pendo.pageLoad() on every route change. Renders nothing, pure side effect.

4 · Tagged elements

  • Stable data-pendo-id hooks on what matters for the funnel: nav-<route> on each sidebar item (app-sidebar.tsx), open-visit (visit-card.tsx), ask-question (ask-question-button.tsx). Intended funnel: the /visits/<visitId> single-visit page.

On load order, if init ever didn't fire: the initialize effect is guarded by if (!PENDO_API_KEY || !window.pendo) return. The agent loads afterInteractive, but the stub window.pendo is created synchronously by the snippet's IIFE, so it exists by the time effects run. If it ever didn't, that ordering is the first place I'd look.

You used Pendo's AI Setup Assistant. Where did you follow it, and where did you diverge?

"I ran Pendo's AI Setup Assistant prompt inside Claude Code for the mechanical install, and it got the framework detection, the loader, the TypeScript types and the environment setup right. I actually improved on it in two places it couldn't know about: the App Router needs the loader in a Next <Script> component rather than the old _document/_app files, and the single-page-app page tracking needs the usePathname plus pageLoad() approach. The one place I diverged on purpose is the visitor identity. In a real install you'd wire the logged-in user straight from your auth provider, somewhere like Clerk gives you useUser(), so Pendo gets the real user ID and email. But this is a product-analytics demo, and if I'm the only real user the dashboard is empty, so I synthesized eight realistic ANZ patients assigned per session instead, so the Paths, Funnels and geo segments all have variety. So the framework and loader were detected; the visitor data I hand-authored for demo realism. That's the judgment call a CE makes by reading the goal, not just running the tool."

Followed the spec (the mechanical layer):

  • Framework detection → Next.js path. Spec splits the snippet into _document/_app; those don't exist in App Router, so the loader went into a <Script strategy='afterInteractive'> component, the correct modern adaptation.
  • Snippet preserved verbatim (Rule 1, never modify the loader).
  • SPA route tracking via usePathname() + pageLoad(), smarter than the generic spec, which wouldn't know the App Router solution.
  • TS types added because tsconfig.json was detected (Rule 4).
  • Env handling using the NEXT_PUBLIC_ prefix Next requires, not the spec's example REACT_APP_ prefix.
  • snake_case mapping (full_name, created_at, is_demo_user) per the spec's normalisation rules.

Diverged on purpose (the data layer):

  • A faithful run would auto-detect the auth provider and map the real authenticated visitor + org. Instead I hand-authored the 8-person roster, flagged is_demo_user / is_demo_account.
  • Init timing follows from that: instead of firing once the real user loads, it fires on an empty-dependency useEffect against a random mock patient, because there's no logged-in user to depend on.
  • Why: a product-analytics demo needs a populated, varied visitor base so Paths, Funnels and geo segments aren't empty. Right for a demo, not the production pattern.

What is the DOM?

"The DOM is the browser's live, in-memory tree of the page. Every element, a div, a button, a table row, is a node in that tree. The HTML is the static source the server sends; the DOM is the living version the browser builds from it, and JavaScript and Pendo read and change the DOM, not the original HTML. That 'live' part matters here because a CRM like the sandbox is a single-page app: when you click Accounts, the content changes without a full page reload, JavaScript just swaps nodes in and out. That's exactly why Pendo needs a page-load signal on route changes, there's no browser navigation event to hang off. Every element has a tag, maybe an id, some classes, optional data attributes and text, and those are the hooks a selector grabs onto."

What's a selector, and why might one break?

"A CSS selector is the address of an element in the DOM. Pendo records a selector when you tag a Feature, and when a user clicks something matching it, Pendo fires a Feature event. The rough specificity order is id beats class beats tag, so you want the most stable, meaningful hook. Selectors break for a few predictable reasons: hashed or auto-generated class names like css-1a2b3c from tools like styled-components or Tailwind, which change every build; position-based selectors like nth-child that break the moment the layout reorders; and ids that include a record number, which won't match the next record. The fix I'd advise a customer's dev team is to add stable, semantic data-pendo-id attributes to the elements that matter. It's a five-minute change, it survives redesigns, and it makes tagging self-documenting, that's the single most valuable thing a CE teaches a new customer's engineers."

What's the difference between Pages and Features?

"Pages are matched by URL rules, like /accounts or /opportunities/*/details, that's where users go. Features are matched by element selectors, that's what users click. Pages are the 'where', Features are the 'what'."

What does the install script actually do?

"It's one JavaScript snippet you drop in the head of the page. It loads the Pendo agent asynchronously so it never blocks rendering. Then you call pendo.initialize with a visitor and an account once you know who the logged-in user is, and a visitor id is mandatory, no id, no events. From that moment the agent passively auto-captures every page view and click and serves Guides, so you don't have to decide in advance what to track. The one distinction worth knowing: the agent captures anything that happens in the UI, client-side, whereas the Track Events API is for things with no UI surface, like a server-side or backend event, which you fire explicitly."

Explain how Pendo captures analytics.

"The key idea is retroactive capture. From the moment the snippet is installed, Pendo records all page and click activity, so when you tag a Feature next month you get historical data going back to the install date. You instrument the app once, then ask questions later. Contrast that with event-based tools like Mixpanel or Amplitude, where you only get data for events you instrumented before the user acted, you have to predict every question up front. That's why a CE can walk into a customer 90 days post-install and immediately produce adoption data on a feature nobody thought to track on day one."

How would you verify a Pendo install is healthy?

"I'd open dev tools and check it three ways. In the Console I'd run a few commands from memory; on the sandbox they return real values. In the Network tab I'd filter for pendo and confirm the agent is loading and the event beacons are firing. And I'd point at the stream of ptm.gif calls, that's the agent shipping page and feature events back to Pendo in real time, that's what 'capturing analytics' looks like on the wire."

A guide isn't appearing. How would you debug it?

"I'd work through it in order."
  1. Is the agent loaded? window.pendo defined, and is the snippet request a 200?
  2. Did initialize run with a real visitor? pendo.getVisitorId(), real id or anonymous?
  3. Is the visitor in the guide's segment? Segments evaluate periodically, not instantly, so a just-published guide or changed segment can lag a few minutes.
  4. Does the activation selector still match? document.querySelector(...), did a hashed class change break the anchor?
  5. Is the guide published and scheduled for now, and has this visitor already dismissed it? (Application/Storage panel.)
  6. Network: is the guide payload loading at all?

Tag a feature on the sandbox. Which selectors would you use, and why?

"Good news, whoever built this gave the primary actions real ids: #add-new and #profile-button, I'd tag straight onto those. But look at these row menu buttons, id=':rj:', that's a React useId, it regenerates every render, so tagging on it would silently break. And the styling is Emotion, so the css- hashes are useless as anchors. This is exactly the conversation I'd have with a customer's dev team: give me a handful of data-pendo-id attributes on the actions that matter and your analytics never break again. I'd also name the Feature by user intent, not button copy, so 'ADD NEW' on /accounts becomes 'Create Account', and if they rename the button my report doesn't break. And I'd tag the Page by URL rule, /opportunities/*/details, not by an element."
