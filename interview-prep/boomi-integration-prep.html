Artur is a Senior SC — a peer, not the HM. He'll respect honesty over polish and smell a bluff instantly. The frame below applies to every answer in this doc.

The 3-part frame for every technical answer

  3. Honest gap — what you haven't done, and how you'd close it.

Length discipline: 60–90 seconds max per answer. If Artur wants depth, he'll ask. Long answers signal nervous over-explanation, not expertise.

02 iPaaS vs ESB vs point-to-point

Foundational. Almost certain to come up.

The plain-English version

PatternWhat it isWhen it winsWhen it fails
Point-to-pointDirect integration between two systems, usually code-level (API → API, file → DB)1–2 integrations, simple, low volumeGrows into spaghetti — N systems means N² connections
ESB (Enterprise Service Bus)Centralised on-prem middleware that routes messages between systems. SOAP-era, heavyweight.Heavy enterprise, regulated, on-prem-firstSlow to change, hard to scale to cloud, vendor lock
iPaaSCloud-native integration platform — low-code canvas, pre-built connectors, runs anywhere (cloud or self-managed atoms)Mixed cloud + SaaS + on-prem stacks, fast time-to-valueNiche batch-heavy or extreme-low-latency cases still need bespoke

Your answer if asked "why has iPaaS won?"

Say: "Three reasons. SaaS exploded, so the stack a customer needs to integrate is no longer all in one data centre — ESBs weren't built for that. Low-code canvases mean a leaner team can own integration, not a dedicated middleware squad. And iPaaS vendors invested in pre-built connectors, which collapses time-to-value from months to weeks. The point-to-point alternative breaks down past a handful of systems."

Your honest experience

Be direct: you've done point-to-point integration (API to API, system to system) and worked alongside customer ERP middleware. You haven't operated a full iPaaS canvas in production yet — that's what Boomi University is closing.

04 API-led vs event-driven

MuleSoft uses "API-led" as marketing. Know the contrast.

API-ledEvent-driven
Architecture organised around well-defined APIs that systems call.Architecture organised around events that systems react to.
Synchronous by default. System A calls System B and waits.Asynchronous by default. System A fires an event, doesn't wait.
Strong when you need request-response (e.g. "give me this customer's order history").Strong when you need real-time fan-out (e.g. "new order placed → notify warehouse, finance, marketing, fraud check").
Familiar to most teams. Easier to debug.Decoupling wins at scale. Harder to debug — observability matters more.
Say: "Most Commercial customers I've worked with are API-led in practice — REST endpoints, synchronous calls, request-response. Event-driven is where they go when scale or decoupling forces it. The two aren't either-or; modern iPaaS supports both, and Boomi can expose any API as an event source via webhooks or trigger flows on events from connected systems."

05 Synchronous vs asynchronous

SyncAsync
Caller waits for the response before continuing.Caller fires the call and continues; response (if any) comes later.
Simple to reason about. Good for UI flows where the user is waiting.Resilient under load. Good for long-running or fan-out work.
Fails fast — if the downstream is down, the user sees it.Handles downstream outages via queue + retry. User isn't blocked.
The Commercial-customer angle: sync is easier; async is where you go when reliability and scale demand it. Most mid-market starts sync and migrates flows to async as volume grows.

09 API Management (APIM) + Federated APIM

JD explicitly calls out Federated APIM. Know what it means.

APIM — what it does

  • Design + publish — version APIs, expose them to internal and external consumers
  • Secure — auth, rate limiting, IP allowlists, threat protection
  • Govern — policies, contracts, SLAs
  • Monetise — usage tiers, billing
  • Observe — usage analytics, error tracking, dev portal

Federated APIM — the Boomi differentiator

Traditional APIM = one central gateway everyone routes through. Brittle at scale, latency issues across regions, sovereignty problems for ANZ customers.

Federated APIM = distributed gateways, central governance. Run a gateway in each region or cloud (AU stays in AU, EU stays in EU), but manage policies, security and observability from one plane.

Say: "Federated APIM is a real win for ANZ Commercial customers — they want data sovereignty without losing central governance. Run a gateway in AU for AU traffic, keep policy management central, get observability across the whole estate. That's a story I've heard from healthcare and finance customers I worked with at inforcer, just framed differently."

11 MDM (Master Data Management) — Boomi DataHub

The problem MDM solves: Same customer has a different record in Salesforce, NetSuite, support tool, marketing tool. Which one is correct? MDM creates a "golden record" by matching, merging and syncing back.

Where it matters in AI

Agents are only as good as the data they query. If the agent calls the CRM and gets a stale customer record, the answer is wrong. MDM is the trust layer that makes AI agents safe to deploy against real customer data.

Say: "DataHub is the layer most Commercial customers don't know they need until they try to deploy AI. The moment an agent gives a customer the wrong order status because there are three contradicting records across systems, the project loses sponsorship. MDM is how you make AI safe in production. I haven't run DataHub specifically — I've worked alongside the problem at Phocas (BI implementations expose data quality issues fast) but that's a learning gap I'd close early."

Why it matters: regulated industries still run massive file-based integrations. iPaaS historically deferred to specialist MFT vendors. Thru brings that capability inside Boomi.

13 Errors, retries, idempotency, dead-letter queues

If Artur probes reliability, these are the four words.

The concepts

  • Retries — when a call fails (network, downstream outage), try again. Usually with exponential backoff so you don't hammer a failing system.
  • Idempotency — the same call run twice produces the same result. Critical for safe retries (you don't want to charge a card twice). Implemented via idempotency keys or natural keys.
  • Dead-letter queue (DLQ) — when a message fails repeatedly, it gets parked in a DLQ instead of blocking the queue. Humans investigate later.
  • Circuit breaker — if a downstream system is failing, stop hammering it for a window so it can recover.
Say: "The reliability story is what separates a demo from production. I've built retry logic and idempotency into the side-project work — the Lumen Health AI Setup Assistant has to re-run safely if a Claude API call fails mid-flow. At enterprise scale, iPaaS like Boomi gives you these as platform primitives — retry policies, DLQs, observability dashboards — instead of every team rebuilding them. That's part of the 'why iPaaS won' argument."

14 Logging, monitoring, observability

  • Logs — events that happened. "Request X failed at step Y."
  • Metrics — numbers over time. Latency, throughput, error rate.
  • Traces — the path of a single request across multiple systems. Critical for debugging distributed flows.

Your angle: at inforcer, observability across the M365 stack (Sentinel for security, Defender for endpoint) was core to the demos. Same principle for iPaaS — Boomi gives you process-level observability so you can debug a failed integration without grepping logs across five systems.

15 MCP (Model Context Protocol) — deeper technical

One-line definition

Open standard (originated by Anthropic) for how AI agents discover and call tools, data sources and prompts. JSON-RPC over stdio or HTTP. Think of it as USB-C for agents.

The architecture

  • MCP host — the AI app (Claude Desktop, an agent in AgentStudio)
  • MCP client — embedded in the host, connects to servers
  • MCP server — exposes tools, resources and prompts. Can wrap any API, database, file system.

Why Boomi cares (and what shipped at World 2026)

Every existing Boomi integration, APIM endpoint or DataHub entity can be exposed as an MCP server. Instantly, 30,000+ customers' existing data and APIs become callable by any agent — governed. This is the through-line: integration becomes the thing that makes enterprise AI safe.

  • AgentStudio exposes 300,000+ endpoints as MCP interfaces — agents reach enterprise systems with no custom integration code per connection.
  • Boomi Connect (now GA) — governed, MCP-based layer connecting AI tools (Claude, Copilot, Gemini) to enterprise apps via 1,000+ managed MCP-enabled tools, plus an MCP Registry to discover/govern MCP servers.
If Artur goes deep: "I've worked with MCP-style patterns on the AI side — the Lumen Health Setup Assistant runs inside Claude Code and uses tool-calling to drive a Next.js build, which is conceptually the same shape. Production MCP servers in an enterprise context I haven't operated — but the pattern is well within the API design and integration work I do. What clicked for me is that by exposing Boomi's 1,000s of connectors as MCP tools, every integration Boomi ever built becomes something an agent can safely call. That's why the message is data activation, not 'we built a chatbot'."

17 Live design question — "How would you integrate X with Y?"

If Artur tests live problem-solving, this is the shape. Use the 5-step frame below.

The 5-step frame (always use this)

  1. Clarify the goal — "What's the business outcome? Is this analytics, operational, real-time, batch?"
  2. Map the data flow — source → transform → target. What's the trigger?
  3. Pick the pattern — sync/async, push/pull, batch/event-driven. Justify in one line.
  4. Auth + governance — how do the systems trust each other? Where does sensitive data sit?
  5. Failure mode — what happens when the downstream is down? Retries? DLQ? Customer-visible degradation?

Worked example: "Integrate Shopify with NetSuite for a mid-market ANZ retailer"

Walk-through: "First I'd clarify — are we syncing orders, inventory, or both, and is it real-time or end-of-day? Assume orders, near-real-time. Pattern: Shopify webhook on order-placed → Boomi process picks it up → transforms to NetSuite sales order schema → POSTs to NetSuite via REST. Auth: OAuth on both sides via the Boomi connectors. Governance: PII (customer email, address) stays in transit only, no persistent storage in the iPaaS layer. Failure: if NetSuite is down, message lands in DLQ, alert fires, replay-safe via idempotency key on the Shopify order ID. Where AgentStudio attaches in 6 months: a customer-service agent that can query order status across both systems via MCP, with the integration as the runway."
Why this answer wins: shows you think like an integration designer, not a feature-pitcher. Lands on Boomi capabilities organically. Closes with the AI expansion story without forcing it.
