1 Connectivity fundamentals — say it simply

Your peer: "important to understand connectivity as a whole — types of connectivity (Database, APIs, ETL)." This is the likely opener.

Integration = getting two systems that weren't built to talk to share data reliably. The main types of connectivity:

TypeWhat it isOne-line example
API / appReal-time request/response between applications over HTTP (REST/SOAP).CRM asks the billing system "what's this customer's balance?" and gets an answer now.
DatabaseReading/writing directly to a DB (SQL queries, stored procs) when there's no API.Pull last night's orders straight out of an on-prem SQL Server table.
ETL / data movementBulk extract → transform → load, usually on a schedule, for analytics or migration.Nightly: pull all sales records, reshape them, load into the data warehouse.
Event / streamingSystems publish events; others react. Real-time, decoupled."Order placed" event fires; inventory, shipping and billing each react.
File / MFTStructured files (CSV, EDI) moved securely between parties — still huge in enterprise.A supplier drops a daily EDI file; you ingest and process it.
The SC framing Artur wants: "Before I pick a connectivity type I ask the customer two things — does this need to be real-time or batch, and is there an API or am I going to the database? That answer picks the pattern. Boomi covers all of them in one platform, which is the whole point — you're not stitching three tools together."

Deeper detail (REST vs SOAP vs GraphQL, ETL vs ELT vs CDC, sync vs async, webhooks vs polling) is in boomi-integration-prep.html §6–10 — reread if Artur pushes past the basics.

3 The Boomi runtime — the bit most candidates miss

Boomi splits into control plane (the cloud platform where you design, deploy, govern and monitor) and the runtime (where integrations actually execute — you place it wherever the data lives).

  • Atom — a single lightweight runtime engine. Run it in Boomi's cloud, your cloud, or on-prem.
  • Molecule — clustered Atoms for HA / load-balancing in production.
  • Atom Cloud — Boomi-hosted runtime for pure cloud-to-cloud, zero infra to manage.
Why this is the killer answer for on-prem: "If the data's on-prem behind a firewall, you drop an Atom inside the customer's network. It makes an outbound call to the Boomi control plane, so no inbound firewall holes — the runtime reaches the on-prem system locally and only the result travels out. Design in the cloud, execute next to the data. That's the architecture that wins security reviews."
Air-traffic-control analogy (from your David prep): control plane = the tower (designs, governs, observes); runtime/Atom = the planes (does the actual flying). Tower never touches the cargo; planes never decide the routes.

Note the 2026 extension: Distributed Agent Runtime applies the exact same "run it on-prem, keep sensitive data behind the firewall" idea to AI agents — same architecture, new layer. Great bridge from fundamentals into the AI story (§4).

6 Live design scenario — "how would you integrate X with Y?"

Peer round with an SC = high chance of a whiteboard-style "walk me through how you'd connect these two systems." Don't jump to a solution — run the SC playbook out loud.

  1. Discover first. "Before I design — real-time or batch? Is there an API on both sides or am I hitting a database? Where do the systems live, cloud or on-prem? What's the data volume and how clean is it?"
  2. Place the runtime. Cloud-to-cloud → Atom Cloud. On-prem system → local Atom behind the firewall (§3).
  3. Pick the pattern. Sync API call vs scheduled ETL vs event-driven — justify it from the answers above.
  4. Map + transform. Boomi's visual mapping between the two schemas; call out data-quality / dedupe if it's master data (→ DataHub).
  5. Reliability. Error handling, retries, idempotency, logging/alerting. (Shows production maturity, not just happy-path.)
  6. Govern + extend. "And because it's on Boomi, that same connection can be exposed as an MCP tool later so an agent could call it — future-proofs the build."
The meta-point Artur is grading: a great SC asks before answering. Talking through discovery questions out loud scores higher than a fast, confident, wrong architecture. Full scenario bank in boomi-integration-prep.html §17.
