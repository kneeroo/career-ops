2.1 · The DOM knock-out criterion

The recruiter literally named "basic concepts about the DOM." Have these crisp:

  • What it is: the browser's live, in-memory tree of the page. Every element — a <div>, a <button>, a table row — is a node. HTML is the static source the server sends; the DOM is the living version the browser builds, and JS (and Pendo) read/change the DOM, not the original HTML.
  • Why "live" matters: this CRM is a single-page app. Click "Accounts" and the URL + content change without a full reload — JS swaps DOM nodes. That's exactly why Pendo needs page-load signals on route changes (your App Router pendo.pageLoad() gotcha): there's no browser navigation event to hang off.
  • Elements have: a tag, an id (unique), classes (reusable), data-* attributes (custom data), text. These are the hooks selectors grab.

2.2 · Selectors — how Pendo finds an element

SelectorExampleMatches
Tagbuttonevery button
Class.btn-primaryelements with that class
ID#save-accountthe one element with that id
Attribute[data-pendo-id="save"]elements with that attribute/value
Descendant.toolbar buttonbuttons anywhere inside .toolbar
Child.toolbar > buttonbuttons directly inside
Positiontr:nth-child(2)the 2nd row

Specificity: id beats class beats tag. Prefer the most stable, specific and human-meaningful hook.

Why selectors break (say this unprompted — the senior signal):

  • Hashed class names — CSS Modules / styled-components / Tailwind JIT emit css-1a2b3c; they change every build, Feature silently stops matching.
  • Position-based (nth-child) breaks the moment layout reorders.
  • Dynamic ids with a record id (#account-4821) won't match the next record.

The fix you'd teach a customer's devs: add stable, semantic data-pendo-id attributes on the elements that matter. Five-minute PR, survives redesigns, self-documents. The single most valuable thing a CE teaches a new customer's engineers.

Pages vs Features: Pages are matched by URL rules (/accounts/*); Features by element selectors. Pages = where users go; Features = what they click.

2.3 · The install script

  • One JS snippet in <head>, loads the agent asynchronously (never blocks the page).
  • Call pendo.initialize({ visitor:{id}, account:{id} }) once you know the user. Visitor id is mandatory — no id, no events.
  • From then the agent passively auto-captures every page view + click, and serves Guides. No need to decide in advance what to track.
  • Snippet vs Track Events API: agent = UI-driven (client-side); Track API = no-UI events (server-side, batch backfills), fired explicitly.

2.4 · The Microdemo concept — capture everything, tag later the differentiator

From install, Pendo records all page + click activity. Tag a Feature next month → you get historical data back to install date. Event-based tools (Mixpanel/Amplitude) only give you data for events you instrumented before the user acted.

"With Pendo you instrument the app once, then ask questions later. With event-based analytics you have to predict every question before you ship. That's why a CE can walk into a customer 90 days post-install and immediately produce adoption data on a feature nobody thought to track on day one."

2.6 · Debugging playbook — "this guide isn't appearing"

  1. Agent loaded? window.pendo defined? Snippet request 200?
  2. Initialize ran with a real visitor? pendo.getVisitorId() — real id or anonymous?
  3. Visitor in the guide's segment? Segments evaluate periodically, not instantly — a fresh publish/segment-change lags a few minutes.
  4. Activation selector still matches? document.querySelector(...) — did a hashed class break the anchor?
  5. Published & scheduled for now? And has this visitor already dismissed it (Storage)?
  6. Network: is the guide payload loading at all?
